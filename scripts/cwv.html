Core Web Vitals

André Schäfer

30. Dezember 2025

Core Web Vitals sind drei von Google definierte Leistungskennzahlen, die messen, wie schnell, reaktionsfähig und visuell stabil eine Webseite für echte Nutzer ist. Die Kennzahlen Largest Contentful Paint (LCP), Interaction to Next Paint (INP) und Cumulative Layout Shift (CLS) bilden seit 2021 einen direkten Rankingfaktor in der Google-Suche. Dieser Lexikon-Eintrag erklärt die drei Metriken im Detail, zeigt dir die wichtigsten Mess-Tools und liefert konkrete Optimierungsansätze für bessere Werte und stabilere Rankings.

Was sind Core Web Vitals?

Core Web Vitals sind eine Teilmenge der Web Vitals — ein von Google entwickeltes Messsystem, das die Nutzererfahrung auf Webseiten quantifizierbar macht. Die drei Kernmetriken bewerten Ladegeschwindigkeit, Interaktivität und visuelle Stabilität einer Seite anhand realer Nutzerdaten.

Google hat die Web-Vitals-Metriken erstmals im Mai 2020 vorgestellt. Seit Juni 2021 fließen sie als Teil des Page-Experience-Signals in das Ranking ein. Der Grundgedanke: Seiten, die schnell laden und stabil dargestellt werden, verdienen eine bevorzugte Position in den Suchergebnissen.

Die Bewertung basiert auf dem 75. Perzentil der gesammelten Felddaten. Eine Seite gilt nur dann als „gut“, wenn mindestens 75 % aller Seitenaufrufe die jeweiligen Schwellenwerte einhalten. Diese Methodik stellt sicher, dass nicht nur der Durchschnitt, sondern die Mehrheit der Besucher eine positive Erfahrung macht.

Neben den drei Kernmetriken erfasst Google weitere ergänzende Messwerte wie Time to First Byte (TTFB), First Contentful Paint (FCP) und Total Blocking Time (TBT). Diese Werte dienen als diagnostische Indikatoren, fließen jedoch nicht direkt in die Core-Web-Vitals-Bewertung ein.

Welche Metriken gehören zu den Core Web Vitals?

Die Core Web Vitals bestehen aus drei Metriken, die jeweils einen zentralen Aspekt der Nutzererfahrung abdecken: Ladeperformance, Reaktionsfähigkeit und visuelle Stabilität.

Metrik Messgröße Schwellenwert (Gut) Auswirkung
Largest Contentful Paint (LCP) Ladezeit des größten sichtbaren Elements ≤ 2,5 Sekunden Wahrgenommene Ladegeschwindigkeit
Interaction to Next Paint (INP) Reaktionszeit auf Nutzerinteraktionen ≤ 200 Millisekunden Gefühlte Reaktionsfähigkeit
Cumulative Layout Shift (CLS) Unerwartete Layout-Verschiebungen ≤ 0,1 Visuelle Stabilität beim Laden

Was misst der Largest Contentful Paint (LCP)?

Der Largest Contentful Paint misst die Renderzeit des größten sichtbaren Inhaltselements im Viewport — typischerweise ein Hero-Bild, eine Überschrift oder ein Textblock. LCP bildet ab, wie schnell der Hauptinhalt einer Seite für den Nutzer sichtbar wird.

Ein LCP-Wert von maximal 2,5 Sekunden gilt als gut, Werte zwischen 2,5 und 4,0 Sekunden erfordern Verbesserungen. Häufige Ursachen für langsame LCP-Werte sind unkomprimierte Bilder, langsame Server-Antwortzeiten und render-blockierende Ressourcen wie CSS oder JavaScript.

Was misst Interaction to Next Paint (INP)?

Interaction to Next Paint erfasst die Latenz zwischen einer Nutzerinteraktion (Klick, Tipp oder Tasteneingabe) und der nächsten visuellen Aktualisierung der Seite. INP hat im März 2024 den bisherigen First Input Delay (FID) als Core-Web-Vital-Metrik abgelöst.

Der entscheidende Unterschied zu FID: INP bewertet nicht nur die erste Interaktion, sondern alle Interaktionen während des gesamten Seitenbesuchs. Der schlechteste gemessene Wert (abzüglich statistischer Ausreißer) bestimmt den INP-Score. Werte unter 200 Millisekunden gelten als gut.

Was misst der Cumulative Layout Shift (CLS)?

Der Cumulative Layout Shift quantifiziert unerwartete Verschiebungen von sichtbaren Elementen während des Seitenaufbaus. Typische Auslöser sind nachladende Werbebanner, Bilder ohne definierte Abmessungen oder dynamisch eingefügte Inhalte.

CLS berechnet sich aus dem Anteil des verschobenen Bereichs multipliziert mit der Verschiebungsdistanz. Ein CLS-Wert von maximal 0,1 gilt als gut. Bewusste Verschiebungen, die durch Nutzeraktionen wie Klicks ausgelöst werden, fließen nicht in die Berechnung ein.

Warum sind Core Web Vitals ein Google-Rankingfaktor?

Core Web Vitals sind ein Rankingfaktor, weil Google die Nutzererfahrung als zentrales Qualitätssignal in seine Suchergebnisse einbezieht. Die Metriken bilden zusammen mit HTTPS, Mobilfreundlichkeit und dem Verzicht auf aufdringliche Interstitials das sogenannte Page-Experience-Signal.

Google bewertet Webseiten nicht ausschließlich nach inhaltlicher Relevanz. Die technische Qualität der Seitenauslieferung spielt eine wachsende Rolle. Eine Seite mit exzellentem Inhalt und schlechten Core-Web-Vitals-Werten kann im Ranking hinter einer ähnlich relevanten Seite mit besserer Performance zurückfallen.

Die Gewichtung ist dabei differenziert zu betrachten. Core Web Vitals funktionieren als Tiebreaker-Signal: Bei annähernd gleichwertiger inhaltlicher Relevanz entscheidet die technische Performance über die Platzierung. Inhaltliche Qualität bleibt der dominante Rankingfaktor — die Performance-Metriken ergänzen ihn.

Für die organische Sichtbarkeit deiner Website bedeutet das: Core Web Vitals allein heben keine schlechte Seite auf Seite 1. Sie können aber den entscheidenden Unterschied zwischen Position 5 und Position 3 ausmachen.

Wie werden Core Web Vitals gemessen?

Core Web Vitals lassen sich mit verschiedenen Google-Tools messen, die entweder Felddaten von echten Nutzern oder Labordaten aus simulierten Tests bereitstellen.

Tool Datentyp Metriken
Google Search Console Felddaten (CrUX) LCP, INP, CLS — gruppiert nach URL-Status
PageSpeed Insights Feld- und Labordaten Alle Core Web Vitals + diagnostische Metriken
Lighthouse Labordaten LCP, CLS, TBT (als INP-Proxy)
Chrome User Experience Report (CrUX) Felddaten LCP, INP, CLS — öffentlich verfügbar via API
Chrome DevTools Labordaten Performance-Panel mit detaillierter Timeline

Die Google Search Console liefert den aussagekräftigsten Überblick, weil sie technische Kennzahlen im Suchmaschinenmarketing direkt mit dem Indexierungsstatus verknüpft. URLs werden dort nach Gut, Verbesserung nötig und Schlecht klassifiziert.

Was ist der Unterschied zwischen Field Data und Lab Data?

Felddaten (Field Data) stammen von echten Chrome-Nutzern und spiegeln die tatsächliche Erfahrung unter realen Bedingungen wider. Google verwendet ausschließlich Felddaten aus dem Chrome User Experience Report für die Ranking-Bewertung.

Labordaten (Lab Data) werden in einer kontrollierten Testumgebung mit standardisierten Bedingungen erhoben — beispielsweise über Lighthouse oder WebPageTest. Sie eignen sich zur Diagnose und Fehlersuche, bilden aber nicht die reale Nutzervielfalt ab.

Die Unterscheidung ist relevant: Lighthouse kann einen perfekten Score anzeigen, während die Felddaten gleichzeitig schlechte Werte melden. Der Grund liegt in Variablen wie Gerätetyp, Netzwerkgeschwindigkeit und geografischem Standort, die nur Felddaten erfassen.

Wie lassen sich Core Web Vitals verbessern?

Die Optimierung der Core Web Vitals erfordert gezielte technische Maßnahmen für jede der drei Metriken. Die folgenden Schritte decken die häufigsten Ursachen für schlechte Werte ab.

  1. LCP durch Bildoptimierung beschleunigen — Komprimiere Bilder in modernen Formaten wie WebP oder AVIF, setze Width- und Height-Attribute und lade Above-the-Fold-Bilder per Preload-Hint priorisiert.
  2. Server-Antwortzeit reduzieren — Ein Time to First Byte unter 800 Millisekunden ist die Basis für gute LCP-Werte. Serverseitiges Caching, ein Content Delivery Network und optimierte Datenbankabfragen senken die Antwortzeit.
  3. Render-blockierende Ressourcen eliminieren — Verschiebe nicht-kritisches JavaScript mit defer oder async. Extrahiere Critical CSS und lade den Rest asynchron nach. Jede eingesparte Blockade verbessert LCP und INP.
  4. JavaScript-Ausführungszeit für INP minimieren — Zerlege lange Main-Thread-Tasks in kleinere Einheiten (Yield-to-Main-Thread-Pattern). Verwende Web Workers für rechenintensive Operationen und reduziere die Gesamtmenge an JavaScript.
  5. Layout-Verschiebungen für CLS verhindern — Definiere feste Abmessungen für Bilder, Videos und Werbeplätze. Reserviere Platz für dynamisch nachladende Elemente und vermeide das nachträgliche Einfügen von Inhalten oberhalb des sichtbaren Bereichs.
  6. Web Fonts stabilisieren — Verwende font-display: swap oder font-display: optional, um unsichtbaren Text während des Schriftladens zu vermeiden. Hoste Schriften lokal statt über externe Dienste und nutze Preload für kritische Font-Dateien.

Bei umfangreicheren technischen Umstellungen ohne Sichtbarkeitsverluste empfiehlt es sich, die Core-Web-Vitals-Werte vor und nach der Änderung systematisch zu dokumentieren.

Wie wirken sich Core Web Vitals auf die Suchmaschinenoptimierung aus?

Core Web Vitals verändern die technische Dimension der Suchmaschinenoptimierung grundlegend. Performance-Optimierung ist nicht mehr nur eine Best Practice, sondern ein messbarer Ranking-Einflussfaktor mit konkreten Schwellenwerten.

Für SEO-Verantwortliche ergibt sich daraus eine erweiterte Aufgabe: Neben Content-Qualität und Backlink-Profil muss die technische Seitenperformance kontinuierlich überwacht und optimiert werden. Die Google Search Console liefert dafür den direkten Zusammenhang zwischen Core-Web-Vitals-Status und dem Prozess, mit dem Google deine Seiten erfasst und verarbeitet.

Ein häufig unterschätzter Zusammenhang besteht zwischen Core Web Vitals und Crawl-Effizienz. Seiten mit guter Performance werden von Googlebot tendenziell häufiger gecrawlt, weil der Server schneller antwortet und weniger Crawl-Budget verbraucht wird. Das beschleunigt die Indexierung neuer Inhalte.

Die Entwicklung geht weiter: Mit dem Aufkommen von KI-gestützten Suchsystemen und ihrer Optimierung gewinnt strukturierte Datenqualität zusätzlich an Bedeutung. Seiten, die sowohl technisch performant als auch semantisch sauber strukturiert sind, haben die besten Voraussetzungen für Sichtbarkeit in klassischen und KI-generierten Suchergebnissen.
